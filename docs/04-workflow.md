# 04. Рабочая петля pair programming с AI

Каждая сессия состоит из 6 шагов. Шаги повторяются: закончил один цикл — начни следующий.

## Схема

![Pair programming workflow](../assets/workflow.svg)

## Шаг 1. Определи задачу

Сформулируй, что нужно сделать, одним предложением.

- ✅ «Добавить сортировку заказов по дате в личном кабинете»
- ❌ «Сделать личный кабинет лучше»

Если не получается одним предложением — задача слишком большая. Разбей.

## Шаг 2. Согласуй план

Попроси агента предложить план до написания кода.

Пример промпта:

```text
Я хочу добавить сортировку заказов по дате в личном кабинете.
Сначала предложи план в 3-5 шагов. Не пиши код, пока мы не согласуем план.
```

Обсуди план, уточни непонятные места, удали лишнее.

## Шаг 3. Реализуй минимальное изменение

Агент пишет код. Ты следишь, чтобы:

- менялось не больше файлов, чем нужно;
- решение оставалось простым;
- не появлялся код «на будущее».

Если агент предлагает сразу много — попроси разбить на части.

## Шаг 4. Проверь

Проверь тремя способами:

1. **Тесты.** Запусти релевантные тесты. Если их нет — проверь вручную.
2. **Diff.** Прочитай, что именно изменилось.
3. **Здравый смысл.** Решает ли изменение задачу? Нет ли побочных эффектов?

## Шаг 5. Закоммить

Сделай коммит с понятным сообщением.

```bash
git add .
git commit -m "feat: sort orders by date in profile"
```

Сообщение должно отвечать на вопрос «что изменилось?».

## Шаг 6. Рефлексия

Задай себе 3 вопроса:

1. Что сработало хорошо?
2. Что можно было сделать быстрее?
3. Что исправить в следующий раз?

Не пиши эссе. Достаточно 2-3 строк в карточке сессии.

---

## Мини-пример

```text
Ты: "Нужно показывать ошибку, если email уже занят при регистрации"
Агент: "План: 1) проверить email в базе, 2) вернуть 409, 3) показать ошибку в UI"
Ты: "Ок, начни"
Агент: [пишет код]
Ты: [читает diff]
Ты: "В UI лучше не показывать сам email в ошибке"
Агент: [исправляет]
Ты: [запускает тесты]
Ты: git commit -m "feat: handle duplicate email in signup"
```

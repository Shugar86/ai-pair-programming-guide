# 05. Среды как инструменты под задачу

**Вердикт:** среда — не религия, а инструмент под задачу.

Среда — не религия, а инструмент. Cursor удобен, когда ты видишь результат. Kimi и Claude хороши для больших проектов и серверов. Copilot и Cody помогают в моменте. Выбирай под задачу, а не под хайп.

Чтобы не путаться в каждом инструменте, ниже одинаковый шаблон:

- **Как показать контекст.** Как дать агенту файлы, карту проекта и `AGENTS.md`.
- **Где смотреть diff.** Где проверять, что агент собирается изменить.
- **Как ограничить scope.** Как удержать агента в нужных файлах.

---

## Cursor

**Когда выбирать.** Ты работаешь с UI, хочешь видеть diff рядом с кодом и быстро кликать «принять» или «откатить».

**Как начать.**

1. Скачай Cursor с [cursor.com](https://www.cursor.com).
2. Открой проект. Нажми `Cmd/Ctrl + L` — откроется чат.
3. Переключись в режим Agent или Composer, если задача трогает несколько файлов.
4. Попроси план до кода.

**Как показать контекст.**

- Используй `@AGENTS.md`, `@README.md`, `@file` или `@folder` в чате.
- В режиме Agent Cursor видит открытые файлы и дерево проекта. Перед сложной задачей открой нужные файлы или явно укажи их через `@`.
- Для общей карты проекта используй `@Codebase` — но проверяй, что он не выдумал структуру.

**Где смотреть diff.**

- Composer/Agent показывает diff перед применением в правой панели.
- Вкладка **Changes** (Ctrl/Cmd + Shift + G) — список всех изменений.
- Включай inline diff в редакторе, чтобы видеть правки построчно.

**Как ограничить scope.**

- Выдели конкретный код перед запросом — Cursor применит правку только к выделению.
- Явно скажи: «Меняй только `ProfileCard.tsx`, не трогай соседние компоненты».
- Отключи авто-применение изменений в настройках, если хочешь сначала читать diff.

## Kimi Code

**Когда выбирать.** Ты в терминале или IDE, проект большой, и тебе нужен агент, который помнит правила из `AGENTS.md`.

**Как начать.**

1. Установи Kimi Code по официальной инструкции для твоего редактора или терминала: [kimi.com/code](https://www.kimi.com/code).
2. Зайди в папку проекта.
3. Создай `AGENTS.md` в корне (шаблон — в [`templates/AGENTS.md`](../templates/AGENTS.md)).
4. Запусти сессию и опиши задачу.

**Как показать контекст.**

- Kimi Code автоматически читает `AGENTS.md` в корне проекта. Убедись, что файл актуален.
- Ссылайся на файлы явно: «Сначала прочитай `src/orders/service.py` и `tests/test_orders.py`».
- Для обзора проекта попроси: «Покажи структуру модулей, которые относятся к задаче».

**Где смотреть diff.**

- Агент выводит diff в терминале перед применением.
- После правки используй `git diff` для финальной проверки.
- В IDE diff отображается в стандартной панели изменений.

**Как ограничить scope.**

- Перечисли разрешённые файлы в первом сообщении: «Работаем только с `src/orders/`».
- Используй selection: выдели фрагмент и попроси изменить только его.
- Правила в `AGENTS.md` работают как защитный барьер: «Меняй не больше 3 файлов без согласования».

## Claude Code

**Когда выбирать.** Ты сидишь в терминале, задача сложная и требует обсуждения. Хорошо работает на сервере и с большими codebase.

**Как начать.**

1. Установи Claude Code с [claude.ai/code](https://claude.ai/code).
2. Запусти `claude` в папке проекта.
3. Опиши задачу текстом.
4. Используй `/clear`, если контекст разросся, и `/help` для команд.

**Как показать контекст.**

- Добавь файлы вручную: `/add src/orders/service.py` или `/add docs/architecture.md`.
- Claude видит рабочую директорию, но не полагайся на это — укажи ключевые файлы.
- В начале сессии скажи: «Начни с `AGENTS.md` и `README.md`».

**Где смотреть diff.**

- Claude показывает diff в терминале перед каждым изменением и спрашивает подтверждение.
- Используй `/git diff` или обычный `git diff`, чтобы сравнить с текущим состоянием.

**Как ограничить scope.**

- Явно укажи границы: «Работаем только в `src/billing/`, API других модулей не меняем».
- Используй `/clear` между несвязанными задачами, чтобы не смешивать контекст.
- Проси план до кода и фиксируй scope в этом плане.

## OpenCode

**Когда выбирать.** Ты хочешь открытое решение, гибкость в выборе модели и интеграцию через MCP.

**Как начать.**

1. Установи OpenCode с [opencode.ai](https://opencode.ai).
2. Настрой провайдера в `opencode.json`.
3. Открой проект и начни диалог.

**Как показать контекст.**

- Подключи `AGENTS.md` и README как источники контекста через MCP или явные ссылки.
- Используй `@file` или аналогичные команды для выбора конкретных файлов.
- Проверь, какой провайдер и модель стоят — от этого зависит размер контекста.

**Где смотреть diff.**

- Diff отображается в UI или терминале в зависимости от провайдера.
- Всегда перепроверяй через `git diff` перед коммитом — OpenCode не единообразен в выводе.

**Как ограничить scope.**

- Ограничивай выбор файлов на уровне UI/CLI: открой только нужную папку.
- В промпте пропиши чёткие границы: «Не меняй конфигурацию, только логику в `src/`».
- Начни с одного провайдера и простого сетапа. MCP и дополнительные модели подключай, когда база работает.

## GitHub Copilot / Cody

**Когда выбирать.** Ты уже в редакторе и нуждаешься в автодополнении, кратком объяснении или мелкой правке прямо в строке.

**Как начать.**

- Copilot: установи расширение в VS Code, JetBrains или Neovim с [github.com/features/copilot](https://github.com/features/copilot).
- Cody: установи расширение Sourcegraph с [sourcegraph.com/cody](https://sourcegraph.com/cody).

**Как показать контекст.**

- Copilot Chat: используй `@workspace` или `@file` для контекста.
- Cody: используй `@codebase` для поиска по проекту и `@file` для конкретных файлов.
- `AGENTS.md` читается не всегда автоматически — скопируй ключевые правила в промпт или приложи файл через `@`.

**Где смотреть diff.**

- Copilot: inline suggestions, панель Copilot Chat с предложениями, diff в редакторе.
- Cody: diff в чате и в панели изменений.
- Для крупных изменений переключайся на инструмент с явным diff-ревью.

**Как ограничить scope.**

- Выдели конкретный код перед запросом — оба инструмента ориентируются на selection.
- Делай запросы короткими и конкретными: «Добавь валидацию email в этой функции».
- Для задач больше одного файла переключайся на Cursor, Kimi или Claude. Не пытайся переписать полпроекта через автокомплит.

## Нюансы по приватности и отправке кода

Код уходит к вендору почти во всех облачных инструментах. Правило простое: не отправляй то, что нельзя показывать.

| Инструмент | Что уходит на сервер | На что обратить внимание |
| :-- | :-- | :-- |
| Cursor | Файлы проекта, diff, история чата | Облачный режим. Не клади в проект `.env`, ключи, пароли, PII. Проверь настройки privacy mode, если работаешь с чувствительным кодом. |
| Kimi Code | Файлы проекта, промпты, diff | Облачный сервис. Избегай отправки секретов и персональных данных. Для корпоративных проектов согласуй политику. |
| Claude Code | Файлы проекта, промпты, diff | Anthropic не использует обычные запросы для обучения моделей по умолчанию, но проверь актуальные условия. Всё равно не отправляй секреты. |
| OpenCode | Зависит от выбранного провайдера | Самый гибкий сценарий: можно настроить локальную модель или self-hosted провайдер. Проверь, куда именно уходят данные. |
| GitHub Copilot | Контекст редактора, часть файлов | Microsoft/GitHub фильтруют секреты, но не полагайся на это. Используй Copilot внимательно с кодом под NDA. |
| Cody | Код, на который ссылаешься | Sourcegraph может работать в облаке или self-hosted. Для приватных репозиториев уточни развёртывание. |

**Универсальные правила:**

- Никогда не отправляй пароли, токены, приватные ключи, личные данные пользователей.
- Для секретного кода предпочитай локальные модели или self-hosted решения.
- Согласуй в команде внутреннюю AI-policy: что можно отправлять агентам, а что — нет.
- Подробнее — в [`docs/19-security-and-policy.md`](19-security-and-policy.md).

## Универсальный вердикт

Не ищи «лучшую» среду. Возьми ту, что уже стоит, и применяй принципы из [`03-principles.md`](03-principles.md). Хороший запрос в подходящей среде даёт больше, чем идеальный инструмент с плохим запросом.

# 06. Промпты как креативные брифы

🌐 [English](en/06-prompt-patterns.md) · **Русский**

**Вердикт:** хороший промпт — это краткий бриф, а не заклинание.

Хороший промпт — не волшебное слово, а краткий бриф. Ты говоришь агенту: что хочу, в каком виде, какие ограничения. Чем точнее бриф, тем меньше доработок.

---

## Классификация паттернов

Каждый паттерн ниже относится к одному из пяти классов. Это помогает быстро выбрать, что использовать под текущую задачу.

| Класс | Когда использовать | Паттерны | Где используется |
| :-- | :-- | :-- | :-- |
| **Plan → Code** | Задача касается нескольких файлов или есть несколько путей решения. | 1. Сначала план, 4. Три варианта, 5. Разбей на шаги | [03-principles.md](03-principles.md), [04-workflow.md](04-workflow.md) |
| **Refactoring** | Нужно изменить структуру кода, не меняя поведение, или сделать точечную правку. | 2. Точечная правка | [10-antipatterns.md](10-antipatterns.md), [11-advanced-scenarios.md](11-advanced-scenarios.md), [18-senior-playbook.md](18-senior-playbook.md) |
| **Diagnostics / Debug** | Что-то сломалось, нужно найти причину и проверить гипотезы. | 6. Стоп и гипотезы | [04-workflow.md](04-workflow.md), [10-antipatterns.md](10-antipatterns.md), [15-troubleshooting.md](15-troubleshooting.md) |
| **Review** | Хочешь проверить diff, найти баги или оценить качество. | 3. Покажи diff перед применением, 7. Ревью как запрос | [04-workflow.md](04-workflow.md), [09-checklists.md](09-checklists.md), [18-senior-playbook.md](18-senior-playbook.md) |

## 1. Сначала план

**Класс:** Plan → Code  
**Где используется:** принцип «сначала план — потом код» в [03-principles.md](03-principles.md), шаг «Наметь путь» в [04-workflow.md](04-workflow.md).

**Когда использовать.** Задача касается больше двух файлов или есть несколько путей решения.

**Пример.**

```text
Я хочу добавить пагинацию в список заказов.
Сначала предложи план в 3–5 шагов. Не пиши код, пока мы не согласуем план.
```

**Почему работает.** Агент фиксирует структуру, прежде чем уходить в детали. Ты ловишь лишний scope до первого diff.

---

## 2. Точечная правка

**Класс:** Refactoring  
**Где используется:** антипаттерн «сделай красиво без границ» в [10-antipatterns.md](10-antipatterns.md), рефакторинг легаси в [11-advanced-scenarios.md](11-advanced-scenarios.md) и [18-senior-playbook.md](18-senior-playbook.md).

**Когда использовать.** Нужно исправить конкретную вещь, не трогая соседний код.

**Пример.**

```text
Сделай минимальное изменение, которое исправляет валидацию email.
Не трогай соседний код и не добавляй ничего "на будущее".
```

**Почему работает.** Ограничение «только эта правка» удерживает агента от рефакторинга, которого никто не просил.

---

## 3. Покажи diff перед применением

**Класс:** Review  
**Где используется:** шаг «Проверь на ощупь» в [04-workflow.md](04-workflow.md), чеклисты в [09-checklists.md](09-checklists.md), системное ревью в [18-senior-playbook.md](18-senior-playbook.md).

**Когда использовать.** Ты не уверен, что агент поймёт задачу с первого раза, и хочешь проверить направление.

**Пример.**

```text
Примени изменение, но сначала кратко объясни, что именно изменится и почему.
```

Можно уточнить уровень:

```text
Объясни на уровне "что увидит пользователь" и "что поменяется в данных".
```

**Почему работает.** Агент вынужден обдумать правку вслух. Ты ловишь странные решения до коммита.

---

## 4. Три варианта

**Класс:** Plan → Code  
**Где используется:** принцип «сначала зачем — потом как» в [03-principles.md](03-principles.md), проектирование сервисов в [04-workflow.md](04-workflow.md).

**Когда использовать.** Есть несколько равных путей, и ты хочешь выбрать осознанно.

**Пример.**

```text
У нас проблема: заказы тормозят при большом списке.
Предложи 3 варианта решения с плюсами и минусами.
Я выберу, и тогда напишем код.
```

**Почему работает.** Агент перестаёт навязывать одно решение. Ты сохраняешь контроль над выбором.

---

## 5. Разбей на шаги

**Класс:** Plan → Code  
**Где используется:** разбиение больших задач в [04-workflow.md](04-workflow.md), принцип KISS и минимальный diff в [03-principles.md](03-principles.md).

**Когда использовать.** Задача размытая вроде «сделай проект» или слишком большая для одной сессии.

**Пример.**

```text
Задача слишком большая: переписать модуль биллинга.
Разбей её на 3–5 маленьких подзадач. Начнём с первой.
```

**Почему работает.** Абстракция превращается в очередь конкретных действий. Каждый шаг можно проверить отдельно.

---

## 6. Стоп и гипотезы

**Класс:** Diagnostics / Debug  
**Где используется:** диагностика в [04-workflow.md](04-workflow.md), игнорирование тестов в [10-antipatterns.md](10-antipatterns.md), разбор ошибок в [15-troubleshooting.md](15-troubleshooting.md).

**Когда использовать.** Вы оба крутитесь вокруг проблемы, ничего не работает.

**Пример.**

```text
Мы застряли с падением сервиса при больших объёмах данных.
Что мы знаем точно, а в чём неуверенность? Какие 3 гипотезы стоит проверить?
```

**Почему работает.** Перебор вариантов останавливается. Вы переходите к проверке фактов, а не к догадкам.

---

## 7. Ревью как запрос

**Класс:** Review  
**Где используется:** чеклисты приёмки в [09-checklists.md](09-checklists.md), системное ревью в [18-senior-playbook.md](18-senior-playbook.md), принцип «проверяй всё» в [03-principles.md](03-principles.md).

**Когда использовать.** Код уже написан — твоим или агентом — и ты хочешь найти слепые зоны.

**Пример.**

```text
Проведи ревью этого кода. Найди:
1. Баги и edge cases.
2. Лишнюю сложность.
3. Нарушения минимального diff.
4. Безопасность: хардкод, утечки, открытые зависимости.
```

Можно добавить контекст:

```text
Представь, что это прод-код в банковском приложении. Будь строгим.
```

**Почему работает.** Агент смотрит на код свежим взглядом и ловит то, что ты пропустил из-за близости к задаче.

## 8. Вердикт первым

**Класс:** Documentation / ADR  
**Где используется:** фиксация решений в [11-advanced-scenarios.md](11-advanced-scenarios.md), проектирование с агентом в [18-senior-playbook.md](18-senior-playbook.md).

**Когда использовать.** Агент пишет длинное вступление, а тебе нужен короткий ответ.

**Пример.**

```text
Дай короткий ответ в первом предложении, потом объясни детали.
```

**Почему работает.** Формат заставляет агента сначала сказать суть, а потом развернуть. Экономит время при быстрых проверках и помогает фиксировать решения в ADR.

## Как комбинировать

Паттерны складываются друг с другом. Не выбирай один — составь бриф из двух-трёх.

Пример для небольшой фичи:

```text
Я хочу добавить сортировку заказов по дате.
Сначала предложи план в 3 шага.
Затем реализуй только первый шаг.
Перед применением объясни, что изменится.
```

Пример для сложной баг-охоты:

```text
Мы застряли с падением сервиса при больших объёмах данных.
Разбей задачу диагностики на подзадачи.
Затем предложи 3 гипотезы и план проверки каждой.
Вердикт давай в первом предложении.
```

Порядок имеет значение: сначала фрейминг задачи, потом ограничения, потом формат ответа.

## Рецепты под стек

Если не хочешь собирать промпт с нуля — возьми готовый бриф для своего стека. В каждом рецепте: роль, ссылка на `AGENTS.md` и первый промпт.

### Next.js / frontend

**Роль:** frontend-имплементер. Пишет UI и клиентскую логику по плану, не лезет в backend.

**AGENTS.md:** шаблон — [`templates/AGENTS.md`](../templates/AGENTS.md), заполни секции стека (Next.js, React, Tailwind, тесты) и правил.

**Первый промпт:**

```text
У нас Next.js 14 проект с TypeScript и Tailwind.
Нужно добавить сортировку заказов по дате в таблице на странице /orders.
Сначала прочитай AGENTS.md.
Предложи план из 3 шагов: API endpoint, компонент таблицы, UI-кнопка.
Не трогай другие страницы и не добавляй новых зависимостей без согласования.
```

### Python backend

**Роль:** backend-имплементер. Делает endpoint'ы, сервисы и тесты, не трогает frontend.

**AGENTS.md:** шаблон — [`templates/AGENTS.md`](../templates/AGENTS.md), заполни стек (FastAPI/Django, PostgreSQL, pytest, ruff/mypy).

**Первый промпт:**

```text
У нас FastAPI + PostgreSQL, тесты на pytest.
Нужно добавить endpoint для экспорта заказов в CSV.
Сначала прочитай AGENTS.md.
Предложи план из 3 шагов: репозиторий, endpoint с streaming response, тесты.
Без pandas и без записи файлов на диск. Не меняй контракты других endpoint'ов.
```

### Infra / DevOps

**Роль:** инженер инфраструктуры. Правит Docker, CI/CD, конфигурацию — сначала план, потом минимальное изменение.

**AGENTS.md:** шаблон — [`templates/AGENTS.md`](../templates/AGENTS.md), заполни стек (Docker, GitHub Actions, Terraform/Compose, prod environment).

**Первый промпт:**

```text
У нас Docker Compose для локальной разработки и GitHub Actions для CI.
Нужно добавить healthcheck для API-сервиса.
Сначала прочитай AGENTS.md и docker-compose.yml.
Предложи план из 3 шагов: Dockerfile healthcheck, compose depends_on condition, CI-шаг проверки.
Не меняй прод-конфигурацию и не трогай секреты.
```

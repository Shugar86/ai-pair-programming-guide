# 08. Упражнения как мини-проекты

🌐 [English](en/08-exercises.md) · **Русский**

Каждое упражнение — маленький проект с измеримым результатом. Ты не просто читаешь правила, а проверяешь их на собственном коде.

> **Вердикт:** теория закрепляется только через руки.

---

## Уровень: Новичок

### Упражнение 1. Минимальный diff

**Уровень:** новичок.  
**Связь:** принцип «Минимальный diff» из [03-principles.md](03-principles.md), шаг «Слепи минимум» из [04-workflow.md](04-workflow.md), ловушка [«Сделай красиво»](10-antipatterns.md) из [10-antipatterns.md](10-antipatterns.md).

**Задача:** добавь поле «Телефон» в форму регистрации и сохрани его в базу данных.

**Что делать:**
1. Попроси агента сделать изменение.
2. Прочитай diff до нажатия Accept.
3. Если агент зацепил лишние файлы — откати и попроси сделать только нужное.

**Ожидаемый исход:** агент вносит ровно одно изменение в один файл; ты читаешь diff до accept и откатываешь всё, что вышло за рамки задачи. Результат — привычка держать сессию внутри одной логической правки.

**Критерий успеха:** в diff ровно один файл и одна логическая правка. Всё остальное — отдельная задача.

**Диалог:**

```text
Ты: Добавь поле phone в форму регистрации.
Агент: [меняет форму, модель, миграцию, тесты]
Ты: Стоп. Сначала только форму и модель. Миграцию и тесты — отдельным шагом.
```

### Упражнение 2. План до кода

**Уровень:** новичок.  
**Связь:** принцип «Сначала план — потом код» из [03-principles.md](03-principles.md), шаг «Наметь путь» из [04-workflow.md](04-workflow.md), ловушка [«Подмена цели задачи»](10-antipatterns.md) из [10-antipatterns.md](10-antipatterns.md).

**Задача:** добавь форму обратной связи на страницу контактов.

**Что делать:**
1. Запрети агенту писать код, пока вы не согласуете план.
2. Согласуй план из 4–5 шагов.
3. Реализуйте только первый шаг.

**Ожидаемый исход:** перед первой строчкой кода у вас есть согласованный план из 3–5 шагов, и в репозитории — один рабочий коммит только по первому шагу. Результат — рефлекс «не кодить вслепую».

**Критерий успеха:** есть письменный план и один рабочий коммит. План лежит в сессии или в `docs/session-notes.md`.

**Диалог:**

```text
Ты: Хочу форму обратной связи. Сначала план из 5 шагов. Код писать не разрешаю.
Агент: 1) добавить endpoint, 2) сделать валидацию, 3) отправлять email, 4) UI, 5) тесты.
Ты: Начни с endpoint и валидации. Остальное — следующие коммиты.
```

## Уровень: Практик

### Упражнение 3. Ревью своего diff'а

**Уровень:** практик.  
**Связь:** принцип «Проверяй всё» из [03-principles.md](03-principles.md), шаг «Проверь на ощупь» из [04-workflow.md](04-workflow.md), ловушка [«Слепое доверие»](10-antipatterns.md) из [10-antipatterns.md](10-antipatterns.md).

**Задача:** найди в своём проекте последний коммит, сделанный с помощью агента.

**Что делать:**
1. Открой diff коммита.
2. Найди 3 вещи, которые можно было бы сделать лучше: лишний код, странное имя, отсутствие тестов.
3. Создай follow-up коммит с исправлениями.

**Ожидаемый исход:** после агентского коммита ты находишь 3 конкретных улучшения и сам делаешь follow-up коммит. Результат — собственное ревью как естественный шаг петли, а не формальность.

**Критерий успеха:** три конкретных замечания и один follow-up коммит.

**Пример замечаний:**

```text
- Имя переменной data ни о чём не говорит.
- Функция делает две вещи: валидацию и сохранение.
- Нет теста на пустой email.
```

### Упражнение 4. Напиши AGENTS.md

**Уровень:** практик.  
**Связь:** принцип «Эскалируй неопределённость» из [03-principles.md](03-principles.md), шаг «Посмотри, что получилось» из [04-workflow.md](04-workflow.md), ловушка [«Диалог без контекста проекта»](10-antipatterns.md) из [10-antipatterns.md](10-antipatterns.md).

**Задача:** создай `AGENTS.md` для своего проекта.

**Что делать:**
1. Возьми шаблон из [`templates/AGENTS.md`](../templates/AGENTS.md).
2. Заполни: язык, стек, правила коммитов, запреты.
3. Проведи одну сессию с агентом, следуя этому контракту.
4. После сессии добавь одно замечание: что в `AGENTS.md` не хватило.

**Ожидаемый исход:** в репозитории появляется живой контракт, по которому проходит одна реальная сессия; после неё добавлено одно уточнение. Результат — умение выращивать долгую память проекта вместо копирования контекста в чат.

**Критерий успеха:** файл `AGENTS.md` в репозитории, один коммит по его правилам и одна правка в `AGENTS.md` по итогам.

## Уровень: Сеньор / Лид

### Упражнение 5. Спроектируй модуль с multi-agent

**Уровень:** сеньор/лид.  
**Связь:** принцип «Сначала зачем — потом как» из [03-principles.md](03-principles.md), шаг «Захвати идею» из [04-workflow.md](04-workflow.md), ловушка [«Подмена цели задачи»](10-antipatterns.md) из [10-antipatterns.md](10-antipatterns.md).

**Задача:** спроектируй небольшой модуль — например, сервис уведомлений или обработчик платежей — с помощью двух агентов. Один агент играет роль архитектора, другой — скептика.

**Что делать:**
1. Попроси первого агента предложить 2–3 варианта архитектуры с плюсами и минусами.
2. Передай эти варианты второму агенту и попроси найти слабые места.
3. Выбери подход и зафиксируй его в ADR через шаблон [`templates/ADR-template.md`](../templates/ADR-template.md).
4. Только после этого разреши писать код.

**Ожидаемый исход:** перед кодом зафиксирован ADR с выбранным подходом, рисками и trade-offs; первая реализация ограничена минимальным скелетом. Результат — архитектурная дисциплина вместо импульсивного кода.

**Критерий успеха:** есть ADR с выбранным подходом и аргументацией, и один коммит с минимальной реализацией.

**Диалог:**

```text
Ты (архитектору): Предложи 3 варианта сервиса уведомлений: email, push, будущие каналы.
Агент 1: Вариант A — отдельные handlers. Вариант B — очередь. Вариант C — pipeline.
Ты (скептику): Вот 3 варианта. Где подвох в каждом?
Агент 2: A плохо масштабируется. B добавляет инфраструктуру. C переусложнён для старта.
Ты: Выбираю A. Фиксирую в ADR. Код — только базовый handler в следующем коммите.
```

## Как использовать в команде

Упражнения работают не только в одиночку. Их удобно разобрать на воркшопе, чтобы выровнять командный стиль работы с агентом.

**Формат:** 1–2 часа, 2–3 упражения, ретро в конце.

**Роли:**
- Фасилитатор — обычно лид или самый опытный в команде. Задаёт темп, показывает diff на экране, сводит к чеклистам.
- Участники — работают парами «человек + агент» в своём редакторе.

**Порядок:**
1. **Разминка — 15 мин.** Все вместе проходят [упражнение 1](#упражнение-1-минимальный-diff) на общем тестовом проекте. Цель — показать, как читать diff до accept.
2. **Основной блок — 40–60 мин.** Каждый выбирает упражнение своего уровня: новички — [упражнение 2](#упражнение-2-план-до-кода), практики — [упражнение 3](#упражнение-3-ревью-своего-diffа) или [4](#упражнение-4-напиши-agentsmd), сеньоры — [упражнение 5](#упражнение-5-спроектируй-модуль-с-multi-agent).
3. **Ретро — 15–20 мин.** Обсуждаем три вопроса:
   - Что сработало сразу?
   - Где diff или план ушли в сторону?
   - Что добавим в `AGENTS.md` или чеклисты команды?

**Материалы:** держи на экране [09-checklists.md](09-checklists.md) и [10-antipatterns.md](10-antipatterns.md) — когда у кого-то застревает сессия, своди его к конкретному пункту или ловушке.

**Итог воркшопа:** каждый уходит с одним конкретным изменением в своём рабочем процессе: кто-то начнёт читать diff, кто-то — писать план до кода, кто-то — вести `AGENTS.md`.

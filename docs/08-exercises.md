# 08. Упражнения как мини-проекты

Каждое упражнение — маленький проект с измеримым результатом. Ты не просто читаешь правила, а проверяешь их на собственном коде.

> **Вердикт:** теория закрепляется только через руки.

---

## Упражнение 1. Минимальный diff

**Уровень:** базовый.  
**Связь:** принцип «Минимальный diff» из [03-principles.md](03-principles.md), шаг «Слепи минимум» из [04-workflow.md](04-workflow.md).

**Задача:** добавь поле «Телефон» в форму регистрации и сохрани его в базу данных.

**Что делать:**
1. Попроси агента сделать изменение.
2. Прочитай diff до нажатия Accept.
3. Если агент зацепил лишние файлы — откати и попроси сделать только нужное.

**Критерий успеха:** в diff ровно один файл и одна логическая правка. Всё остальное — отдельная задача.

**Диалог:**

```text
Ты: Добавь поле phone в форму регистрации.
Агент: [меняет форму, модель, миграцию, тесты]
Ты: Стоп. Сначала только форму и модель. Миграцию и тесты — отдельным шагом.
```

## Упражнение 2. План до кода

**Уровень:** базовый.  
**Связь:** принцип «Сначала план — потом код» из [03-principles.md](03-principles.md), шаг «Наметь путь» из [04-workflow.md](04-workflow.md).

**Задача:** добавь форму обратной связи на страницу контактов.

**Что делать:**
1. Запрети агенту писать код, пока вы не согласуете план.
2. Согласуй план из 4–5 шагов.
3. Реализуйте только первый шаг.

**Критерий успеха:** есть письменный план и один рабочий коммит. План лежит в сессии или в `docs/session-notes.md`.

**Диалог:**

```text
Ты: Хочу форму обратной связи. Сначала план из 5 шагов. Код писать не разрешаю.
Агент: 1) добавить endpoint, 2) сделать валидацию, 3) отправлять email, 4) UI, 5) тесты.
Ты: Начни с endpoint и валидации. Остальное — следующие коммиты.
```

## Упражнение 3. Ревью своего diff'а

**Уровень:** базовый.  
**Связь:** принцип «Проверяй всё» из [03-principles.md](03-principles.md), шаг «Проверь на ощупь» из [04-workflow.md](04-workflow.md).

**Задача:** найди в своём проекте последний коммит, сделанный с помощью агента.

**Что делать:**
1. Открой diff коммита.
2. Найди 3 вещи, которые можно было бы сделать лучше: лишний код, странное имя, отсутствие тестов.
3. Создай follow-up коммит с исправлениями.

**Критерий успеха:** три конкретных замечания и один follow-up коммит.

**Пример замечаний:**

```text
- Имя переменной data ни о чём не говорит.
- Функция делает две вещи: валидацию и сохранение.
- Нет теста на пустой email.
```

## Упражнение 4. Напиши AGENTS.md

**Уровень:** продвинутый.  
**Связь:** принцип «Эскалируй неопределённость» из [03-principles.md](03-principles.md), шаг «Посмотри, что получилось» из [04-workflow.md](04-workflow.md).

**Задача:** создай `AGENTS.md` для своего проекта.

**Что делать:**
1. Возьми шаблон из [`templates/AGENTS.md`](../templates/AGENTS.md).
2. Заполни: язык, стек, правила коммитов, запреты.
3. Проведи одну сессию с агентом, следуя этому контракту.
4. После сессии добавь одно замечание: что в `AGENTS.md` не хватило.

**Критерий успеха:** файл `AGENTS.md` в репозитории, один коммит по его правилам и одна правка в `AGENTS.md` по итогам.

## Упражнение 5. Спроектируй модуль с multi-agent

**Уровень:** продвинутый.  
**Связь:** принцип «Сначала зачем — потом как» из [03-principles.md](03-principles.md), шаг «Захвати идею» из [04-workflow.md](04-workflow.md).

**Задача:** спроектируй небольшой модуль — например, сервис уведомлений или обработчик платежей — с помощью двух агентов. Один агент играет роль архитектора, другой — скептика.

**Что делать:**
1. Попроси первого агента предложить 2–3 варианта архитектуры с плюсами и минусами.
2. Передай эти варианты второму агенту и попроси найти слабые места.
3. Выбери подход и зафиксируй его в ADR через шаблон [`templates/ADR-template.md`](../templates/ADR-template.md).
4. Только после этого разреши писать код.

**Критерий успеха:** есть ADR с выбранным подходом и аргументацией, и один коммит с минимальной реализацией.

**Диалог:**

```text
Ты (архитектору): Предложи 3 варианта сервиса уведомлений: email, push, будущие каналы.
Агент 1: Вариант A — отдельные handlers. Вариант B — очередь. Вариант C — pipeline.
Ты (скептику): Вот 3 варианта. Где подвох в каждом?
Агент 2: A плохо масштабируется. B добавляет инфраструктуру. C переусложнён для старта.
Ты: Выбираю A. Фиксирую в ADR. Код — только базовый handler в следующем коммите.
```

# 09. Чеклисты как pre-flight checks

Эти чеклисты — не правила, которые надо соблюдать. Это быстрые проверки перед взлётом. Пробежался по пунктам — и полетел.

> **Вердикт:** чеклист экономит тебе силы, а не добавляет работы.

---

<a id="before-session"></a>

## Перед сессией

Проверь, что ты готов к диалогу. Если хоть один пункт минимального набора не проходит — сузь задачу.

### Минимальный набор

- [ ] Задача сформулирована одним предложением.
- [ ] Ясно, зачем это нужно и что изменится после.
- [ ] Знаю, как проверить результат: тесты, ручной сценарий или оба.
- [ ] В этой сессии решаю не больше одной задачи.
- [ ] Ветка в git чистая или я знаю, куда коммитить.

### Расширенный набор

- [ ] `AGENTS.md` и README актуальны.
- [ ] Есть контекст от предыдущей сессии: сессионная карточка или заметки.
- [ ] Определён критерий готовности: что именно считать концом задачи.
- [ ] Понятно, кто будет принимать или ревьюить результат.
- [ ] Учтены риски и чувствительные данные — см. [19-security-and-policy.md](19-security-and-policy.md).
- [ ] Есть план B, если агент уйдёт в сторону: откат, сужение scope, новый промпт.

**Если не проходит:** вернись к [03-principles.md](03-principles.md), раздел «Сначала зачем — потом как».

<a id="before-accept"></a>

## Перед принятием diff

Не жми Accept, пока не посмотрел diff. Это твоя страховка.

### Минимальный набор

- [ ] Я прочитал весь diff, не только первые файлы.
- [ ] Каждое изменение относится к текущей задаче.
- [ ] Нет кода «на будущее».
- [ ] Имена переменных и функций говорят сами за себя.
- [ ] Нет хардкода секретов, паролей и токенов.

### Расширенный набор

- [ ] Я понимаю, почему агент сделал именно так.
- [ ] Публичные контракты, API и инварианты не изменены без явного запроса.
- [ ] Запущены релевантные тесты, линтер и проверка типов.
- [ ] Новые зависимости согласованы и нужны прямо сейчас.
- [ ] Учтены очевидные edge-cases и пути отката.
- [ ] Я могу объяснить каждое изменение вслух.

**Если не проходит:** попроси агента объяснить строчку или откати изменения.

<a id="before-commit"></a>

## Перед коммитом / PR

Коммит — точка сохранения. Убедись, что в неё попало только то, что нужно.

### Минимальный набор

- [ ] Тесты проходят или есть запись, почему их нет.
- [ ] Линтер и форматтер не ругаются.
- [ ] Сообщение коммита описывает суть, а не просто «правки».
- [ ] В коммит не попали `.env`, ключи, локальные настройки IDE.
- [ ] Diff соответствует сообщению коммита.

### Расширенный набор

- [ ] Ручной сценарий проверен: фича работает не только в теории.
- [ ] Документация и комментарии обновлены, если менялся контракт.
- [ ] Коммит маленький: 1–3 логически связанных файла.
- [ ] CI зелёный или объяснено, почему это нормально.
- [ ] Назначен ревьюер или проведён self-review.
- [ ] Откат коммита безопасен: нет разрушительных миграций и односторонних контрактов.

**Сообщение хорошее:** `feat: add phone field to registration form`.  
**Сообщение плохое:** `fix`.

<a id="lead-review"></a>

## Для лида / ревьюера

На уровне команды важно смотреть не только на строки, но и на то, как решение вписывается в систему.

### Минимальный набор

- [ ] Задача вписывается в текущую архитектуру и ADR.
- [ ] Контракты, API и инварианты не нарушены.
- [ ] Решение не перегружено: нет лишних абстракций и кода «на потом».
- [ ] Есть план тестирования и внедрения.
- [ ] Риски отката и побочные эффекты учтены.

### Расширенный набор

- [ ] Проведено системное ревью: границы модулей, инварианты, откаты.
- [ ] Проверены кросс-командные зависимости и контракты между сервисами.
- [ ] Учтены метрики, мониторинг и логирование.
- [ ] Решение соответствует внутренней AI policy и требованиям безопасности.
- [ ] Обновлён `AGENTS.md` или ADR, если появились новые соглашения.
- [ ] Ретро-заметка добавлена: что сработало, что улучшить.

<a id="stop-signals"></a>

## 🚩 Стоп-сигналы — повод остановиться и пересобрать

Если замечаешь что-то из этого — не продолжай в том же темпе. Остановись, откати или перефразируй задачу.

- [ ] В diff внезапно десятки файлов вместо 1–2.
- [ ] Появились вызовы функций или методов, которых нет в проекте или библиотеке.
- [ ] Агент меняет контракты функций без явного запроса.
- [ ] В коде много «на будущее» без связи с задачей.
- [ ] Тесты не запускались после изменений.
- [ ] Я сам не могу объяснить, что произошло в коммите.

Что делать: откатись к чистой ветке, сузь задачу и начни с нового плана. Подробнее о ловушках — в [10-antipatterns.md](10-antipatterns.md).

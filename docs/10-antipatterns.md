# 10. Антипаттерны как творческие ловушки

Когда ты создаёшь быстро, легко свалиться в одну из этих ловушек. Они не значат, что ты плохо работаешь. Они значат, что скорость обогнала фокус.

> **Вердикт:** ловушка — не приговор. Узнаёшь по симптому, вытаскиваешь по правилу.

---

## 1. «Сделай красиво» без границ

**Симптом:** задача звучит расплывчато, агент устраивает массовый рефакторинг по всему репозиторию.

**Что ломается:** в diff десятки файлов, половина из них не относится к задаче, тесты падают, поведение меняется там, где не просили.

**Как ловить:**
- Формулируй задачу одним предложением.
- Добавь явные границы: «только этот файл», «не трогай UI», «не меняй архитектуру».
- Проси план до кода и согласовывай scope.

**Пример промпта:**

```text
Сделай код чище только в файле src/orders/service.py.
Не меняй публичные контракты и не трогай другие модули.
Сначала покажи план из 3 шагов.
```

## 2. Невидимые изменения

**Симптом:** агент тихо правит типы, сигнатуры функций, контракты API или инварианты доменной модели.

**Что ломается:** код компилируется, но логика едет. Ошибки всплывают позже, когда уже не помнишь, что менялось.

**Как ловить:**
- Читай diff целиком, а не только то, что агент выделил.
- Проси объяснить, какие контракты изменились, до применения.
- Запускай тесты после каждого шага.

**Диалог:**

```text
Ты: [смотрит diff] Ты поменял сигнатуру createUser. Это не обсуждали.
Агент: Это нужно было для новой валидации.
Ты: Откати сигнатуру. Сделай валидацию внутри, не меняя контракт.
```

**Пример промпта:**

```text
Я вижу в diff, что ты поменял сигнатуру createUser. Мы это не обсуждали.
Откати сигнатуру. Сделай валидацию внутри, не меняя публичный контракт.
```

## 3. Галлюцинации API и SDK

**Симптом:** в коде появляются методы, флаги или библиотеки, которых не существует в используемой версии.

**Что ломается:** сборка падает, а если не падает — runtime падает в проде.

**Как ловить:**
- Проверяй любой незнакомый метод по официальной документации.
- Указывай версию библиотеки в промпте.
- Не разрешай агенту подключать новые зависимости без согласования.

**Пример промпта:**

```text
Используй только методы из Django 4.2, описанные в официальной документации.
Если нужен метод, которого нет в этой версии — скажи об этом.
```

## 4. Подмена цели задачи

**Симптом:** вместо простого скрипта внезапно рождается мини-фреймворк с конфигами, плагинами и абстракциями.

**Что ломается:** решение перегружено, его сложно поддерживать, а исходная задача решается черезчур.

**Как ловить:**
- Фиксируй критерии успеха: «что должно работать после этого коммита?»
- Спрашивай: «это нужно для текущей задачи?»
- Используй YAGNI как щит.

**Диалог:**

```text
Агент: Я добавил плагиновую систему для обработки ошибок.
Ты: Нам нужна только одна обработка 404. Удали плагины, сделай один handler.
```

**Пример промпта:**

```text
Нам нужна только одна обработка 404. Удали плагиновую систему и сделай один handler.
```

## 5. Псевдо-рефакторинг ради рефакторинга

**Симптом:** агент переименовывает переменные, переставляет импорты и меняет форматирование, не улучшая архитектуру.

**Что ломается:** diff шумный, ревью превращается в ад, история коммитов теряет смысл.

**Как ловить:**
- Разделяй рефакторинг и фичи по разным коммитам.
- Проси агента не менять стиль, если задача не про это.
- Используй линтер/форматтер, чтобы стиль не обсуждался.

**Пример промпта:**

```text
Не переименовывай переменные и не меняй форматирование.
Только структурные изменения по текущей задаче.
```

---

## 6. Игнорирование тестов и инструментов

**Симптом:** агент пишет код, но не запускает тесты, не проверяет типы, не смотрит на линтер.

**Что ломается:** код «выглядит ок», но не работает или не проходит CI.

**Как ловить:**
- Всегда проси запустить релевантные тесты.
- Проверяй вывод CI/линтера перед коммитом.
- Не принимай изменения, если не видел зелёных проверок.

**Пример промпта:**

```text
После изменений запусти тесты для модуля orders.
Если падает — объясни причину и предложи фикс. Не проси меня проверять вслепую.
```

## 7. Диалог без контекста проекта

**Симптом:** ты копипастишь куски кода в чат, не объясняя архитектуру, и получаешь решение, которое не вписывается в проект.

**Что ломается:** агент предлагает «как в туториале», а не «как у вас в команде».

**Как ловить:**
- Дай агенту [`AGENTS.md`](../templates/AGENTS.md), README и структуру проекта.
- Указывай существующие соглашения: «у нас для этого используем сервисный слой».
- Показывай примеры похожего кода из проекта.

**Диалог:**

```text
Ты: У нас валидация проходит через сервисный слой, а не в сериализаторе.
Агент: Понял. Перенесу логику в services/validation.py.
```

**Пример промпта:**

```text
У нас валидация проходит через сервисный слой, а не в сериализаторе.
Перенеси логику в services/validation.py, не меняя подход проекта.
```

## 8. Слепое доверие

**Симптом:** ты жмёшь «Accept all», не читая diff, потому что «агент же умный».

**Что ломается:** в код попадает всё, что угодно: баги, уязвимости, лишние зависимости, изменения, которых не просил.

**Как ловить:**
- Читай diff перед каждым accept.
- Задай правило: «я не коммичу то, что не понимаю».
- Используй `git diff` и откатывай, если что-то не так.

**Диалог:**

```text
Ты: [смотрит diff] Здесь ты добавил зависимость left-pad. Зачем?
Агент: Для форматирования строки.
Ты: У нас уже есть utils/string.js. Удали зависимость и используй его.
```

**Пример промпта:**

```text
[смотрит diff] Здесь ты добавил зависимость left-pad. У нас уже есть utils/string.js.
Удали left-pad и используй utils/string.js.
```

## 🚩 Стоп-сигналы — триггер для паузы

Если видишь это — остановись. Не пытайся «добить» задачу в том же направлении. Откатись, пересобери план и начни снова.

- В diff внезапно десятки файлов вместо 1–2.
- Появились вызовы функций/методов, которых нет в проекте или библиотеке.
- Агент меняет контракты функций без явного запроса.
- В коде много «на будущее» без связи с задачей.
- Тесты не запускались после изменений.
- Ты сам не можешь объяснить, что произошло в коммите.

Каждый стоп-сигнал — повод вернуться к [04-workflow.md](04-workflow.md) и пройти ещё один цикл: идея, план, минимум, проверка, рефлексия.

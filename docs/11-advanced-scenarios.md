# 11. Продвинутые сценарии работы с AI

Базовые принципы работают везде. Но в сложных задачах — легаси, миграции, проектирование, несколько проектов — им нужна оркестрация. Ты держишь карту, агенты двигают фигуры.

> **Вердикт:** чем сложнее задача, тем больше планирования и тем меньше кода за один раз.

---

## Сценарии по типам

| Тип | Когда применяется | Главная сложность |
| :-- | :-- | :-- |
| Рефакторинг легаси | Старый код работает, но никто не понимает, как | Страх сломать то, что «и так работает» |
| Cross-module changes | Изменение трогает несколько модулей | Скрытые зависимости и контракты |
| Несколько репозиториев / workspace | В работе 2–3 проекта сразу | Контекст растворяется между стеками |
| Распределённая команда | Люди в разных часовых поясах, агент — общий язык | Непрерывность контекста между сессиями |

---

## 1. Рефакторинг легаси-модуля

Старый код работает, но никто не понимает, как именно. Страшно трогать.

### Мульти-сессионный flow

#### День 1 — понимание и карта

**Цель:** узнать, что модуль делает, где швы, где риски.

**Промпт:**

```text
Проанализируй файл legacy_billing.py.
Не меняй код. Составь список:
1) что делает этот модуль,
2) какие функции можно вынести отдельно,
3) где скрытые зависимости.
```

**Артефакты:**
- `docs/adr/ADR-011-legacy-billing-boundaries.md` — границы модуля и первые швы.
- `docs/notes/legacy-billing-map.md` — карта входов, выходов, побочных эффектов.
- Сессионная карточка: что узнали, какие риски.

#### День 2 — первый шов

**Цель:** вынести одну чистую функцию без изменения поведения.

**Промпт:**

```text
Вынеси расчёт налога из legacy_billing.py в отдельный helper.
Не меняй логику. Только дублируй поведение в новом файле и замени вызов.
После правки запусти тесты.
```

**Артефакты:**
- Новый `src/billing/tax_helper.py`.
- Обновлённый `legacy_billing.py` с импортом.
- Сессионная карточка: diff, тесты, замечания.

#### День 3 — следующий шов и фиксация

**Цель:** вынести вторую функцию, зафиксировать intermediate state.

**Промпт:**

```text
Теперь вынеси расчёт скидок по той же схеме.
После этого обнови ADR: какие функции уже изолированы, что остаётся внутри модуля.
```

**Артефакты:**
- Обновлённый ADR.
- `AGENTS.md` проекта дополнен: правило «рефакторинг легаси — один шов за сессию».
- Коммит: `refactor: isolate tax and discount helpers from legacy billing`.

**Правило:** сначала понимание, потом разделение, потом изменение. Никогда всё сразя.

## 2. Cross-module changes

Изменение API в одном месте ломает контракты в других. Нужно видеть цепочку.

### Мульти-сессионный flow

#### День 1 — карта зависимостей

**Цель:** найти всех потребителей изменяемого контракта.

**Промпт:**

```text
Мы планируем изменить формат ответа /api/v1/orders.
Найди все места в проекте, которые используют это поле.
Для каждого места: файл, функция, как используется. Не меняй код.
```

**Артефакты:**
- `docs/adr/ADR-012-orders-response-format.md` — зачем меняем, какие модули затронуты.
- Список потребителей с оценкой риска.

#### День 2 — адаптеры и тесты

**Цель:** внедрить новый формат так, чтобы старые потребители не сломались.

**Промпт:**

```text
Добавь адаптер, который принимает новый формат /api/v1/orders
и отдаёт старый интерфейс для модулей X и Y.
Напиши тесты на оба формата.
```

**Артефакты:**
- `src/orders/adapter.py`.
- Тесты на совместимость.
- Сессионная карточка: что проверяли.

#### День 3 — миграция потребителей

**Цель:** перевести один модуль на новый формат.

**Промпт:**

```text
Переведи модуль src/reports/ на новый формат ответа orders.
Удали адаптер для этого модуля, если он больше не нужен.
Запусти тесты reports и orders.
```

**Артефакты:**
- Обновлённый `src/reports/`.
- ADR дополнен: статус миграции, что осталось.
- Коммит: `feat: migrate reports to new orders response format`.

**Правило:** cross-module изменение — это не один PR. Это цепочка адаптеров и тестов.

## 3. Несколько репозиториев / workspace

Когда в работе 2–3 репозитория, контекст растворяется. Агент начинает путать стеки, соглашения и границы.

### Мульти-сессионный flow

#### День 1 — workspace-контекст

**Цель:** зафиксировать, как проекты связаны и какие контракты между ними.

**Промпт:**

```text
У нас три репозитория: project-a (API), project-b (frontend), project-c (worker).
Составь список контрактов между ними: endpoint'ы, очереди, общие схемы.
Для каждого контракта — где источник истины.
```

**Артефакты:**
- `workspace-context.md` на уровень выше репозиториев.
- `AGENTS.md` в каждом проекте.

#### День 2 — изменение в project-a

**Цель:** внести изменение так, чтобы project-b и project-c узнали о рисках.

**Промпт:**

```text
Перед правкой проверь workspace-context.md.
Мы меняем формат ответа API в project-a. Какие endpoint'ы в project-b и project-c его используют?
Не пиши код, только список мест и риски.
```

**Артефакты:**
- Обновлённый `workspace-context.md` с пометкой об изменении.
- Сессионная карточка: кого нужно предупредить.

#### День 3 — синхронизация и PR

**Цель:** подготовить согласованные изменения в project-b или project-c.

**Промпт:**

```text
В project-b нужно адаптировать вызов /orders под новый формат.
Используй workspace-context.md и ADR из project-a.
Не меняй ничего, что не относится к этому контракту.
```

**Артефакты:**
- PR в project-a с ADR.
- PR в project-b с адаптацией.
- Сессионная карточка: зависимости между PR.

**Правило:** проекты не должны догадываться друг о друге в голове агента. Контекст — в файлах.

**Пример структуры:**

```text
~/workspace/
  workspace-context.md
  project-a/
    AGENTS.md
  project-b/
    AGENTS.md
  project-c/
    AGENTS.md
```

## 4. Распределённая команда

Люди в разных часовых поясах. Сессии разрываются. Нужна непрерывность.

### Мульти-сессионный flow

#### День 1 — задача и план

**Цель:** зафиксировать задачу и план так, чтобы следующий человек продолжил без переспросов.

**Промпт:**

```text
Мы начинаем задачу: добавить rate limiting в API.
Сначала предложи план из 3 этапов. Каждый этап — рабочее состояние.
Запиши его в docs/tasks/rate-limiting-plan.md.
```

**Артефакты:**
- `docs/tasks/rate-limiting-plan.md`.
- `docs/adr/ADR-013-rate-limiting.md` — если архитектура нетривиальна.
- Сессионная карточка: что решено, кто продолжает.

#### День 2 — реализация первого этапа

**Цель:** внедрить базовый rate limiter, не ломая остальное.

**Промпт:**

```text
Реализуй этап 1 из docs/tasks/rate-limiting-plan.md.
Используй AGENTS.md проекта.
После правки: тесты, diff, обнови план — отметь, что сделано.
```

**Артефакты:**
- Рабочий код первого этапа.
- Обновлённый план.
- Сессионная карточка: что работает, что блокирует.

#### День 3 — интеграция и ретро

**Цель:** закрыть задачу и оставить заметку для команды.

**Промпт:**

```text
Заверши этап 3. Запусти полный набор тестов.
После этого напиши короткую ретро-заметку в docs/retro/rate-limiting.md:
что сработало, что можно было быстрее, что передать следующей смене.
```

**Артефакты:**
- Коммит(ы) по задаче.
- Ретро-заметка.
- Обновлённый `AGENTS.md`, если появились новые правила.

**Правило:** смена заканчивается не кодом, а записью. Следующий человек начинает с файла, а не с расспросов.

## Multi-agent и оркестрация

Одному агенту сложно держать в голове и код, и архитектуру, и ревью. Раздели роли между несколькими агентами или сессиями.

**Роли:**
- **Архитектор** — обсуждает подход и фиксирует ADR.
- **Имплементер** — пишет код по плану.
- **Ревьюер** — проверяет diff, ловит проблемы.

`AGENTS.md` становится контрактом: в нём прописано, кто за что отвечает и какие инструменты использовать.

**Конкретный пример стека:**
- **Claude Code** как архитектор — проектирует подход, фиксирует ADR.
- **Cursor** как имплементер — пишет код по плану.
- **Kimi Code** как ревьюер — проверяет diff и ловит проблемы.

**Пример промпта для ревьюера:**

```text
Сейчас ты — строгий ревьюер.
Не пиши код. Прочитай этот diff и найди:
1) нарушения минимального diff,
2) потенциальные баги,
3) несоответствие плану.
```

**Правило:** каждый агент держит одну роль. Не проси архитектора править опечатки, а ревьюера — проектировать.

## ADR-шаблон для фиксации решений

Когда архитектура нетривиальная, зафиксируй решение. Используй шаблон [`templates/ADR-template.md`](../templates/ADR-template.md).

AI ускоряет механическую работу, но не отменяет твоей ответственности за архитектуру и последствия.

## Сеньорский взгляд

> **Вердикт:** на сложных задачах ты перестаёшь быть заказчиком кода и становишься архитектором системы.

Проектируй с агентом, а не через агента. Попроси сгенерировать варианты, найти trade-offs, зафиксируй ADR — и только потом переходи к коду. Разделяй роли: один агент держит архитектуру, другой пишет, третий ревьюит. Это не лишние рты — это разные режимы внимания, которые редко умещаются в одной сессии.

Контекст живёт в файлах, а не в чате. `AGENTS.md`, `workspace-context.md` и сессионные карточки — твоя долгая память между проектами и сессиями. Переключаясь между репозиториями, не тащи контекст с собой: закрой одну сессию, открой другую, дай ссылки на нужные файлы.

Подробнее — в [18-senior-playbook.md](18-senior-playbook.md).

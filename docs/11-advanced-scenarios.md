# Продвинутые сценарии работы с AI

Здесь собраны сценарии, которые выходят за рамки «дописать функцию» или «пофиксить баг». Они полезны, когда ты уже уверен в базовых принципах pair programming и хочешь использовать агент как усилитель, а не как автогенератор шаблонного кода.

## 1. Рефакторинг легаси-модуля

**Контекст:** есть старый, запутанный модуль, который все боятся трогать. Тестов мало или нет, документация устарела.

**Как использовать AI:**
- начать с картирования: попросить агента описать, какие входы/выходы у модуля, какие зависимости и побочные эффекты;
- вместе с агентом найти естественные «швы» для разделения на части (по домену, по слоям, по типу ответственности);
- предложить несколько вариантов последовательного рефакторинга с минимальным риском (например, вынести одну чистую функцию, добавить тесты к ней, потом двигаться дальше).

**Ключевой момент:** не давать агенту «переписать всё с нуля». Вместо этого использовать его как аналитика и генератора небольших безопасных шагов.

## 2. Миграция с одной технологии на другую

**Контекст:** нужно перейти, например, с одного фреймворка на другой, сменить ORM, перенести логику в новый сервис.

**Как использовать AI:**
- попросить агента сравнить старый и новый стек в разрезе твоего проекта (роутинг, модели данных, авторизация, деплой);
- составить вместе «карта миграции»: какие модули трогаем, какие остаются нетронутыми, какие нужно написать заново;
- генерировать адаптеры и временные прослойки, чтобы старый и новый код могли жить рядом;
- использовать агента для массовых, но однотипных изменений (правка конфигов, миграция схем, механическая замена паттернов).

**Ключевой момент:** сначала проектирование стратегии, потом автоматизация рутинных частей, а не наоборот.

## 3. Работа с чужим монолитом

**Контекст:** ты приходишь на проект или открываешь учебный репозиторий, где огромный монолит без понятной структуры.

**Как использовать AI:**
- начать с обзора: попросить агента построить карту модулей, найти точки входа и центральные зависимости;
- поощрять его создавать краткие текстовые описания: «что делает этот модуль», «как течёт данные через систему»;
- использовать агента как навигатор: «покажи все места, где используется эта сущность/функция»;
- постепенно собирать живую документацию в виде markdown-файлов, а не держать всё только в чате.

**Ключевой момент:** AI помогает строить ментальную модель проекта, но решения о том, куда лезть и что трогать, принимаешь ты.

## 4. Проектирование нового модуля или сервиса

**Контекст:** ты ещё не пишешь код, а решаешь, как должна выглядеть новая часть системы.

**Как использовать AI:**
- проговорить с агентом требования и ограничения: что важно по перфомансу, безопасности, масштабируемости;
- попросить несколько разных архитектурных эскизов (вариантов), а не один «идеальный»;
- обсудить плюсы и минусы каждого варианта для твоего конкретного случая;
- зафиксировать выбранный вариант в виде короткого ADR (Architecture Decision Record) и только потом переходить к коду.

**Ключевой момент:** использовать агента как собеседника для архитектурных решений, а не только как генерилку классов.

## 5. Multi-agent и оркестрация

**Контекст:** ты используешь Cursor Agents, OpenClaw или другую систему, где несколько агентов решают разные задачи.

**Как использовать AI:**
- явно разделять роли: один агент отвечает за архитектуру, другой — за реализацию, третий — за тесты и проверку инвариантов;
- фиксировать общие правила в `AGENTS.md`: что им делать можно, а чего нельзя (не трогать инфраструктуру, не менять схемы БД без плана миграции и т.п.);
- использовать одного агента как «ревьюера» для патчей, которые сгенерировал другой;
- не забывать, что ты всё равно остаёшься единой точкой ответственности.

**Ключевой момент:** несколько агентов усиливают друг друга только тогда, когда у них есть понятные границы и общий контекст, который контролируешь ты.

## 6. Поддержка и эволюция учебных проектов

**Контекст:** курсовые, дипломы, лабораторные, которые живут больше одного семестра и передаются от группы к группе.

**Как использовать AI:**
- вести с агентом живую историю проекта: что добавили, какие решения приняли и почему;
- просить его помогать в подготовке документации для следующих студентов: шаги запуска, структура репозитория, типичные баги;
- использовать агента для адаптации задач под новый стек или новые требования, не убивая при этом дух исходного проекта.

**Ключевой момент:** AI помогает сохранить и передать накопленное понимание проекта, а не только «дописать ещё один модуль».

**Scope:**

2. Дополнить `docs/11-advanced-scenarios.md` разделом «Сеньорский взгляд».

---

## 1. VibeSpark

- **Кто читает:** сеньоры, техлиды, архитекторы, которые уже работают с агентами и хотят масштабировать это на команду/систему.
- **Боль:** базовые принципы понятны, но непонятно, как агентов встраивать в проектирование, ревью, миграции и управление несколькими проектами.
- **Ощущение после:** «Теперь я вижу, как агенты становятся частью моей архитектурной практики, а не просто ускорителем кодинга».

## 2. Вайб-токен

**`systems-thinker`**

Тон: опытный напарник, который думает на уровне системы. Не учителя, не гуру. Конкретные паттерны, которые можно взять и применить.

## 3. Правила голоса для senior-слоя

1. **Без повторения базы.** Не объясняем KISS и minimal diff заново — ссылаемся на 03.
2. **Системные примеры.** Не «добавь кнопку», а «выдели сервис уведомлений из монолита».
3. **Trade-offs в центре.** Каждое решение — с плюсами, минусами и условиями применимости.
4. **Опыт, не теория.** Паттерны из реальной работы с агентами и из agent-os.

## 4. Содержание `docs/18-senior-playbook.md`

### 1. Проектирование с агентом
- Как использовать агента на этапе проектирования, не только имплементации.
- Паттерн: «архитектор + скептик» — два агента оспаривают решение.
- ADR через диалог с агентом.
- Пример: проектирование сервиса уведомлений.

### 2. Мульти-агентные сетапы
- Роли: архитектор, имплементер, ревьюер, тестировщик.
- Когда распараллеливать, а когда нельзя.
- Контракты между агентами: `AGENTS.md`, ADR, чеклисты.
- Пример: рефакторинг модуля с тремя агентами.

### 3. Управление контекстом между сессиями
- `AGENTS.md` как долгая память.
- Карточки сессий.
- Передача контекста между Cursor, Kimi, Claude.
- Пример: возвращаешься к проекту через неделю — что показать агенту.

### 4. Управление несколькими проектами
- `workspace/` или `docs/` как единый источник истины.
- Разделение контекста по проектам.
- Горячее переключение между проектами.
- Пример: правишь баг в одном проекте, а агент помнит контракт второго.

### 5. Ревью на уровне системы
- Как попросить агента проверить не код, а архитектуру.
- Чеклист системного ревью: границы модулей, контракты, инварианты, откаты.
- Пример: агент находит нарушение инварианта в новой фиче.

### 6. Рефакторинг легаси под контролем
- Стратегия: карта → швы → адаптеры → постепенный перенос.
- Использование агента для механических изменений, не для архитектурных решений.
- Пример: выносим расчёт цен из монолита.

### 7. Оценка качества агентского вывода
- Как понять, что агент сделал хорошо, а не просто «выглядит ок».
- Метрики: покрытие тестами, сложность diff, соответствие плану, отсутствие галлюцинаций.
- Пример: агент предложил 5 файлов — как оценить, можно ли принять.

### 8. Когда не использовать агентов
- Исследование неизвестной области.
- Критичные архитектурные решения без данных.
- Ситуации, где ответственность непереносима.
- Пример: агент предлагает менять БД — почему это стоп-сигнал.
